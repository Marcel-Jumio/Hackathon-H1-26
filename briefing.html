2-day Hackathon · H1 / 2026

Turn any customer URL into an on-brand IDV demo.

Paste a prospect's website → get a shareable microsite that mimics their onboarding, with a live, brand-themed Jumio WebSDK embedded at the identity-verification step. Built for sales & pre-sales to lower the barrier to attention.

🎯 Pre-sales attention-grabber 🤖 Auto / semi-auto brand capture 🎨 Real WebSDK theming 🌍 Distributed, mixed-skill team

The ideaWhy this wins attention

In sales/pre-sales the hardest thing is getting a prospect to care. A generic demo says "here's our product." A microsite that looks like their own onboarding says "here's you, with friction removed." It closes the imagination gap instantly.

The problem

Cold demos don't land. Prospects can't picture the SDK inside their own flow, and a real integration takes an engineering team days.

The solution

A one-click "showcase creator": capture a brand's look & feel → auto-theme the WebSDK → embed it in a brand-skinned onboarding microsite → share a link.

The "wow"

"Watch us do in 10 seconds what normally takes an integration team a week." It's real — the SDK is genuinely token-themeable.

How it worksOne artifact ties it all together

Everything hangs off a single file in the middle — brand-profile.json. The harvesting team produces it; the theming engine consumes it; the microsite and UI read it. Lock this schema in the first two hours and every team can build in parallel against mock data — exactly what a distributed, mixed-skill team needs.

The contract — brand-profile.json

// The single source of truth. Harvest fills it, the engine themes from it.
{
  "brand": {
    "name": "Acme Bank",
    "sourceUrl": "https://acme.com",
    "logo": { "light": "<url|base64>", "dark": "<url|base64>" },
    "colors": { "primary": "#0a3", "onPrimary": "#fff",
                 "background": "#fff", "text": "#111", "accent": "#0a3" },
    "typography": { "family": "Inter", "url": "https://fonts.googleapis.com/..." },
    "radius": "12px"
  },
  "journey": {
    "vertical": "banking",
    "steps": ["Create account", "Verify identity", "Done"],
    "idvStepIndex": 1,
    "ctaLabel": "Verify your identity"
  },
  "sdk": { "workflow": "id+selfie", "locale": "en" }
}

The mapping layer (brand → SDK) is the secret sauce and reuses what the WebSDK already supports: colors.primary → a generated lightness ramp on --jumio-sdk-color-primary1..7; logo → the <template id="jumio-logotype-*"> slots; typography → a font <link> + --jumio-sdk-font-family.

ScopeProtect an "always-green" demo

The riskiest part (auto-crawling) must never break the demo. Build it as a ladder where every rung is independently demoable.

Must · Day 1

Manual path

Fill the JSON by hand → press Generate → themed SDK in a brand-skinned microsite → shareable link. This alone is a complete, winning demo.

Should · Day 2 AM

Semi-auto

Crawler pre-fills most of the JSON, a human nudges a few fields, then Generate. Best balance of wow and reliability.

Could · Stretch

Full-auto

URL in → JSON out → microsite, zero manual edits, for 2–3 hand-picked "hero" brands.

Why this works for a distributed, mixed-skill team

  • Contract-first → parallelism. Once WP2 ships the schema + mock JSONs (~hour 2), WP3/4/6 each work against mocks with zero dependency on the crawler or each other.
  • Mock-first → no blocking handoffs. Nobody waits for WP1. The crawler can land on Day 2 and just swap real JSON in for mock JSON.
  • The JSON is the handoff token. A region going offline hands off by committing an updated brand-profile.json + a one-line note — no live sync needed.
  • Low-tech tracks are real work. Manual brand harvesting (WP1) and demo/hero-brand prep (WP7) are on the critical path for a convincing demo and need a browser, taste and domain sense — not code.

Risks…and cheap mitigations

Crawler is flaky / sites block it

Manual edit is always the fallback. Pre-cache the hero brands' JSON so the live demo never depends on a network crawl.

Auto-extracted colors look ugly

The mapping generates a sensible ramp from one primary color; the human nudges one value if needed.

CORS / fonts / logo on dark vs light

Server-side fetch for assets; Google-Fonts match with system fallback; light/dark logo variants are already solved in the SDK.

Scope creep on "mimic their flow"

Don't clone their site. A clean, brand-skinned 3-step stepper reads as "their flow" and is achievable. Say this out loud as a team.

Open decisionsLet's settle these first

  • Team & timezones — who's in, and in which regions? (Drives the real handoff chain.)
  • Hero brands — which 2–3 recognizable brands do we guarantee look great?
  • Build target — a new little app inside the WebSDK monorepo (apps/showcase-studio), or a standalone app that imports the published @jumio/websdk?
  • Crawl strategy — client-side fetch vs a small server-side extractor (avoids CORS, more reliable).
