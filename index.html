1 Problem & Impact

What problem are you solving?

Cold identity demos don't land. Prospects can't picture the Jumio WebSDK inside their own onboarding flow, and a real branded integration takes an engineering team days. That friction kills momentum in the earliest, most fragile part of a deal.

Who is the target user?

Jumio field teams — AEs, solutions engineers and BDRs running first-touch outreach, discovery and demos. Secondary: the prospect on the other side, who sees their own brand reflected back on a live Jumio flow.

Why is this problem important?

Attention is the scarcest asset in the field. A generic demo says "here's our product." A microsite skinned as the prospect's own onboarding says "here's you, with friction removed" — it shifts the conversation from features to fit, and frames Jumio as identity intelligence built into their journey rather than a bolt-on check.

How do users currently solve this?

Generic, unbranded demo flows, or slide mockups of "what it could look like." A truly branded demo today requires a real integration — days of engineering the field can't requisition for a single conversation.

What measurable impact could this create?

Branded demo turnaround cut from days of engineering to under a minute, self-serve, directly inside the Studio. Leading indicators to track post-launch: number of demo links published per rep, demo-to-opportunity conversion lift, and SE/AE hours saved per demo.

2 Solution & Innovation

Your solution in one sentence?

The Jumio SmartDemo Tool takes a prospect's brand — from a manually-curated preset today, with automatic URL-based capture as a near-term next step — auto-themes the real Jumio WebSDK to match, and produces a shareable, brand-skinned onboarding microsite with a live verification flow embedded, in seconds.

What makes it unique or innovative?

It drives the WebSDK's full theming surface — 342 --jumio-sdk-* design tokens plus ~45 template slots — from a single generated brand profile, instead of the handful of branding knobs a normal integration exposes. One profile in → a fully repainted, working SDK out, with per-screen copy/string overrides and custom guidance text on top.

Why is it better than alternatives?

It's real, not a mockup — <jumio-sdk> is genuinely live and runs an actual ID Check / Selfie / Liveness workflow against real Jumio credentials. And it's self-serve: a field rep with API credentials goes from "paste a URL" to a published, shareable link with no code and no engineering ticket.

Most creative aspect?

The color-ramp generator in engine/brand-to-theme.mjs: one brand primary hex expands into a 7-step lightness ramp (color-primary1..7) that drives every interactive state of the SDK. Paired with the publish pipeline, which mints a fresh single-use SDK token at publish time and stores the rendered microsite as a private Vercel Blob behind a slug URL — turning a "demo" into a real, time-boxed, securely shareable link.

If you had another month?

Harden brand harvesting/analyze for arbitrary sites, add dark-mode token twins, expand journey templates beyond the current verticals, and add a hosted gallery of saved / published demos with open-rate and completion analytics.

3 Technical Execution

Walk us through the architecture.

A single contract — brand-profile.json — connects every component. The Studio (React/Vite, studio/App.jsx) lets a user pick or generate a brand profile, configure product/locale/credentials, and edit it live. The theming engine (engine/brand-to-theme.mjs) maps it to --jumio-sdk-* CSS variables and an HTML snippet (logo, fonts, color ramp, template overrides). The microsite renderer (microsite/render.mjs) assembles a full standalone page — a brand-skinned stepper with <jumio-sdk> mounted at the IDV step. A set of Vercel serverless functions (/api) handle the Jumio OAuth2/account/workflow flow for live previews and the publish pipeline, which mints a single-use SDK token, stores the rendered profile + token as a private Vercel Blob, and serves it back via a /studio/:slug rewrite.

Technologies, frameworks, APIs?

React 19 + Vite 8 for the Studio; @jumio/websdk (vendored locally at install time) for the embedded verification flow; @vercel/blob for storing published microsites; sonner for UI feedback; and plain .mjs Vercel serverless functions calling Jumio's OAuth2 token endpoint, account/workflow-execution creation, and PREPARED_DATA upload APIs (region-aware: amer-1/emea-1/apac-1 → us/eu/sgp).

Most technically challenging part?

The publish pipeline. A published demo link needs a working <jumio-sdk> session embedded in a page that can be opened standalone, days later — but Jumio SDK tokens are short-lived/single-use and credentials must never reach the client. We solved this by minting a fresh single-use token at publish time, storing it server-side in a private blob, and rendering the page on demand at request time via /api/site, with expiry enforced. The color-ramp generator — turning one brand hex into an accessible 7-step ramp that holds up across 342 tokens — was the other major piece.

Reliability / scalability / security?

API credentials are used per-request only and never persisted — only the resulting short-lived/single-use SDK token is stored, and it's stored in a private blob. Published links carry an expiresAt and the site endpoint enforces it server-side, so stale links fail closed. Token lifetime is validated and clamped (5 min – 60 days). All brand-profile fields are HTML-escaped before injection into the rendered microsite. The whole thing is stateless serverless functions + blob storage, so it scales horizontally with no shared server state.

Trade-offs you made?

Contract-first (brand-profile.json + JSON Schema) so the engine, microsite, and Studio could be built and iterated independently. A curated set of pre-built brand profiles (Bybit, Revolut, LinkedIn, Xbox, G2, Okada) over a fully general crawler — a reliable Quick Load beats a flaky live crawl during a pitch. We iterated twice on the publish token strategy (reuse session token → mint a fresh single-use token at publish) and chose the more secure option despite the extra API call. No automated test suite given the time budget — verified via repeated builds and manual live-preview testing.

[Insert demo video — placeholder link in the header.]

Key feature that delivers the most value?

The combination of the theming engine + live preview: a brand profile turns into a fully re-skinned, running <jumio-sdk> instantly — the moment a prospect sees their own brand wrapped around a real verification flow — paired with publish, which turns that moment into a link the rep can actually send. For non-technical roles this is the whole win: an AE or BDR with no code and no engineering ticket can build and ship a fully branded, working demo themselves, in the time it takes to prep for a call — turning "let me get engineering to mock that up" into "here, click this link."

Who uses it, and how? (by role)

One tool, three field plays — full breakdown in the use-case cards below. In short: the AE opens a discovery call with the prospect's own branded flow; the SE skips the integration ask and tailors a live demo on the fly; the BDR drops a personalized branded link into cold outreach to earn the reply.

How does a new user get started?

Enter Jumio API credentials (key, secret, region, workflow key) once in the credentials bar. From there, Quick Load presets give an immediately working branded demo with zero setup, so a new user can explore customization and publishing before ever building a profile from scratch. Full reset gives a clean slate at any point. No install, no code.

Assumptions validated during the hackathon?

That the WebSDK is richly themeable from CSS custom properties and template slots alone (342 tokens / ~45 slots), that a single primary brand color is enough to derive a coherent full theme via an automated color ramp, that a debounced live preview with a real mounted SDK stays responsive, and that a single-use-token-minted-at-publish + private-blob pattern is enough to ship secure, shareable demo links without a database.

How would you measure success at launch?

Adoption — demo links published per rep per week, distinct brand profiles created. Engagement — open rate on published links, completion of the embedded IDV step, and expiry-before-open rate. Time-to-demo — minutes from "paste URL" to "published link" vs. the days a hand-built branded demo takes today. Longer-term: demo-to-opportunity conversion lift attributable to branded links.

5 Team & Learning

Teng Tjun (Singapore) built the theming engine that converts brand JSON into live WebSDK styling, and framed and developed the microsite infrastructure on Vercel, as well as providing the serverless API call infrastructure alongside local storage logic into Studio UI. Mike, Ian, and Abby (UK, US EST, and PT timezones) owned the review and delivery of the pitch. Sovio created the email template to connect Studio and Microsite. Abby (US PT) owned the core messaging woven through the solution overview and demo guide.

Obstacles, and how you overcame them?

The hard constraint was a distributed, mixed-skill team working across time zones with only two days. We locked in the technical requirements early and basically stayed with the initial brand scheme without changing it down the line, which set the tone and pace for everything that followed. The follow-the-sun model turned the time zone spread into near-round-the-clock progress, with a short end-of-day handoff note from each region. Scope — the line between what we would ship and what we would roadmap — stayed blurry longer than it should have. But the largest obstacle was organizational: a top-down culture of "you have to" rather than "organise yourself." A hackathon, like a barcamp, works bottom-up, thriving on intrinsic motivation that converts to company value. Control is not the answer. We overcame it through something you cannot plan: engagement, communication, humour, and mutual respect. We created the framework and resisted the urge to micromanage. Once teams owned their work and felt trusted, execution became highly effective.

What did you learn?

The biggest lesson was that a strong contract is more valuable than constant coordination. By defining a single JSON file as the source of truth between teams, we enabled parallel workstreams without requiring continuous synchronization. While we still held a few alignment meetings, the architecture allowed teams to move independently and efficiently. Crucial was the intelligence gathered from the web-sdk-playground, which the web team uses to have "all info websk" in one place. This formed the basis and was identified BEFORE the actual hackathon. We also learned a lot about onboarding non-technical contributors. AI tools have created new opportunities for people outside traditional engineering roles to meaningfully participate in the development of solutions. Once the team had a clear understanding of how to leverage their skills and divide responsibilities, collaboration accelerated, and execution became highly effective.

What would you do differently?

From a technical perspective, we took the exact right choices in everything we chose to use. Take more time upfront to understand each team member's strengths before dividing the work. We came together well, but a clearer picture of everyone's skills from the start would have helped us assign the right work to the right people faster and get more out of the time we had. We also would have drawn the line between "what we ship" and "what we roadmap" earlier, so we could focus on nailing the core demo rather than chasing scope we did not have time to finish.

How it worksOne artifact ties it all together

Everything hangs off a single file — brand-profile.json. The harvesting team produces it; the theming engine consumes it; the microsite and UI read it. Lock this schema in the first two hours and every team builds in parallel against mock data.

The contract — brand-profile.json

// The single source of truth. Harvest fills it, the engine themes from it.
{
  "brand": {
    "name": "Acme Bank",
    "colors": { "primary": "#0a7d33", "onPrimary": "#fff",
                 "text": "#111", "pageBackground": "#fff" },
    "fontFamily": "Inter",
    "logo": "<url|base64>"
  },
  "journey": { "steps": ["Create account", "Verify identity", "Done"],
              "idvStepIndex": 1 },
  "sdk": { "workflow": "id+selfie", "locale": "en" }
}

The mapping layer (brand → SDK) reuses what the WebSDK already supports: colors.primary → a generated lightness ramp on --jumio-sdk-color-primary1..7; logo → the <template id="jumio-logotype-*"> slots; fontFamily → a font <link> + --jumio-sdk-font-family.
